授权码模式回顾

授权码模式(authorization code)是功能最完整、流程最严密的授权模式, 它的特点就是通过 客户端的后台服务器 与 "服务提供商"的认证服务器 进行互动 ; 所以之前我们在介绍OAuth的时候, 其实主要叙述的就是这种比较常用的客户端授权模式 ;

运行步骤回顾

1.客户端将用户导向认证服务器的授权页面
此时也就是客户端请求认证服务器的授权页面 ;

2.用户在授权页面给予客户端授权(authorization grant)后, 认证服务器会生成一个授权码code, 并带着这个code重新导向客户端事先指定的 “重定向URL” ;

3.客户端的回调接口收到授权码, 开始在客户端的服务器中(注意这里不是通过浏览器了)去直接向认证服务器申请令牌, 此时是带着客户端收到的授权码去申请令牌的;

4.认证服务器最终会返回令牌(access_token);

5.客户端使用令牌去 资源服务器 申请用户信息;

第三阶段就是使用access_token获取用户数据

这一过程中涉及了不少敏感参数和数据, 例如client_secret相当于是第三方应用自己的密码, access_token某种程度上来讲就是用户的session id, 由于这些参数以及数据极其特殊, 我们当然得确保它们的安全性, HTTPS加密传输以及安全存储是必不可少的防护手段, 不过仅仅做到这些是远远不够的, 因为其实在这个流程里存在一个弱点， 容易被攻击者利用进行CSRF攻击, 稍后讲解 ;
