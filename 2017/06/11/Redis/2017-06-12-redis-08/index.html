Redis事务与单线程

  1. 因为Redis是单线程的, 所以即使多个客户端同时来对同一数据发来很多命令, 也会被串行挨个执行;

    • 但是对于事务, 因为它最终提交(exec)才会算做被执行, 所以假设有多个客户端A发起事务(其中要修改数据A)
    • 在客户端的事务没有最终exec之前, 客户端B是可以直接操作数据A的, 因为串行化只是串行化 执行 这个动作, 它把整个事务当成了一个比较慢的命令, 既然你慢, 那其他快的就先执行;
    • 但可悲的是, 事务虽然慢, 但是在事务开始前, 它也是满足了 和 客户端B 执行命令一样的条件(比如都发现库存还有一件), 这就出错了

  2. 和mysql的srialize隔离级别一样, 即使是串行执行命令, redis也逃不过高并发事务时的 更新丢失 问题;

    • mysql是使用乐观锁, 悲观锁来解决的 参考MySQL高并发事务问题 及 解决方案
    • 而redis的事务也是通过与WATCH(乐观锁)的结合才得以解决这个问题

Redis事务与WATCH

  1. 演示Redis事务在客户端高并发时出现的 丢失更新 问题

  2. redis 为了解决高并发事务时这种 丢失更新 的问题, 提供了 WATCH

WATCH介绍

  1. redis 并没有实现典型的加锁功能(比如MySQL中, 在访问以写入为目的数据时 SELECT FOR UPDATE), 因为加这种悲观锁可能会造成长时间的等待;

  2. 所以redis为了尽可能地减少客户端的等待时间, 采用了WATCH监控机制, 如果某个客户端A在事务开始之前 WATCH 了一个key, 那么其实就相当于对该key加了乐观锁

    • 事务中正常执行你要执行的操作 (乐观地认为不会有其他客户端抢在你前面去改动那个key)
    • 直到当客户端A真正exec的时候, 才会验证客户端A之前WATCH(监控)的key是否被变动过, 如果变动过, 则客户端A可以进行重试;
