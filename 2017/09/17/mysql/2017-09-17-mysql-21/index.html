21. MySQL 高并发事务问题

2017-09-17
  • MySQL

前言

上一篇MySQL事务简介中对MySQL事务的相关基本概念及特性做了简单描述, 但仅仅了解这些还是不够的, 在实际场景中, 你可能经常会面对一些高并发应用, 此时就需要进一步了解高并发下事务中的一些常见问题;

高并发事务问题

在并发量比较大的时候, 很容易出现 多个事务并行 的情况;
假设有两个事务正在同时进行, 值得注意的是: 它们两者之间是互相不知道对方的存在的, 各自都对自身所处的环境 过分乐观, 从而并没有对自己所操作的数据做一定的保护处理, 所以 最终导致了一些问题的出现;

脏读

  1. Mysql中, 如果 事务A 读取了另一个并行 事务B 未最终提交的写数据, 那事务A的这次读取操作就叫 脏读

    • 因为 事务A 此时读取到的是 尚未被持久化的数据 (事务B中修改的数据还不具备事务的 持久性 特性)

    • 事务A 此时读取的数据也叫 脏数据
      事务B 最终可能会因为内部其他后续操作的失败或者系统后续突然崩溃等原因, 导致事务B最终整体提交失败而回滚, 那么最终 事务A 拿到的自然就是 脏的数据 了;

  3. 解决方案 : RC+

    • 在MySQL中, 事务已经用自身特性 隔离性 的 – READ COMMITED或以上隔离级别 解决了这个问题;
    • READ COMMITED 隔离级别保证了: 在事务中, 某条语句执行前, 已经被其他事务提交的数据, 对该语句都是可见的;

不可重复读

  1. 现在, 上面的 脏读问题 已经被解决了, 那就意味着事务中的每条语句读取到的数据都是 持久性 的数据(被别的事务最终 提交/回滚 的具有持久性的落地数据);

  2. 但脏读问题的解决, 也仅仅只能保证你在事务中每次读到的数据都是持久性的数据而已;

  3. 试想, 如果在一个事务中多次读取同一个数据, 正好在两次读取之间, 另外一个事务已经完成了对该数据的修改并提交, 那问题就来了: 两次读取的结果不一样了

  4. 多次读取数据不一致, 会导致什么问题呢? 一个事务中为什么要多次读取同一数据, 什么场景下需要这么做? 如下汇款案例:

  5. 解决方案 : RR+

    • 在MySQL中, 事务已经用自身特性 隔离性 的 – REPEATABLE READ或以上隔离级别 解决了这个问题;

    • REPEATABLE READ 级别保证了:
      在事务中, 某条语句执行前, 已经被其他事务 提交/回滚 的落地数据, 对该语句都是可见的; (READ COMMITED)
      在事务中, 多次读取同一个数据(在两次读取操作之间, 无论数据被 提交/回滚 多少次(即无论落地过多少遍), 每次读取的结果都应该是和事务中第一次读取的结果一样;

幻读

  1. 可以参考 MySQL官方文档对 Phantom Rows 的介绍 )

  2. 不可重复读 和 幻读 这两个概念经常搞混

    • 但 不可重复读 主要是说多次读取同一条记录, 发现该记录中某些列值被其他事务修改过;
    • 而 幻读 主要是说多次读取一个范围内的记录(包括直接查询所有记录结果或者做聚合统计), 发现结果不一致(比如发现增加/减少了一条记录);

  3. 解决方案: RR + MVCC

    • 其实对于 幻读, 在Mysql的InnoDB存储引擎中, 事务默认的 RR 级别已经通过 MVCC机制 帮我们解决了(并非完全解决), 所以该级别下, 你也模拟不出幻读的场景; 退回到 RC 隔离级别的话, 你又容易把 幻读 和 不可重复读 搞混淆, 所以这可能就是比较头痛的点吧!
    • 另外可以参考《高性能MySQL》对 RR 隔离级别的描述

      理论上, RR级别是无法解决幻读的问题, 但是由于InnoDB引擎的RR级别还使用了MVCC, 所以也就避免了幻读的出现!

更新丢失

最后聊一下 高并发事务 的另一个问题: 丢失更新问题, 该问题和之前几个问题需要区分开, 因为解决方案不是一类!

第一类丢失更新

  1. A事务撤销时, 把已经提交的B事务的更新数据覆盖了

  2. 需要注意的是: 这种情况在Mysql中不会出现, 因为即使是InnoDB事务的最低隔离级别(READ UNCOMMITED), 由于会对写数据设置 x 锁;
    这样, 事务A想对数据做读取/修改, 也都只能要等到事务B的数据修改完毕并提交( 释放了x锁才行 );
    这样之后, 即便回滚, 也回滚到的是事务B已经最终提交的那次改变;

第二类丢失更新

  1. A事务 覆盖 B事务 已经提交的数据，造成B事务所做操作丢失

  2. 这种丢失更新, 是比较常见的, 是无法依靠前三种隔离级别来解决的, 而且可以看到事务A不是要将数据更新为最终的一个确认值, 而是依据读取到的值进行计算后写入的, 这样必然会导致事务B的更新丢失

  3. 解决方案: 可以使用如 乐观锁(在修改时, where判断数据是否为你读取时的数据; 或者提供数据版本字段来控制), 悲观锁 来解决;

注意

  1. 注意: 最高隔离级别 Serializable 在实际应用场景中并不被采用 (并且, 它也无法解决更新丢失的问题)

  2. 因为虽然serialize将事务串行化了, 但貌似只是将同时到达的多个不同事务中的语句按照锁进行了串行化, 虽然读写都会互相阻塞(经过测试), 但是这只是代表事务之间需要等待, 但最终都会执行, 所以还会造成超卖!;

  3. 所以即使是最高隔离级别, 也还是会出现 第二类更新丢失 的高并发事务问题:
